# 開発ワークフロー

このドキュメントでは、開発の基本的な手順と規約について詳しく説明します。

---

## 🔴 最重要: ローカル完結型開発フロー

**⚠️ 重要**: このプロジェクトでは**ローカル完結型**の開発フローを採用します。**GitHubへのプッシュは不要です。**

### 方式の概要

| 項目 | 従来の方式 | 本プロジェクトの方式 |
|------|----------|-------------------|
| PR作成 | GitHubでPR作成 | ローカルでレビュードキュメント作成 |
| プッシュ | 必須 | **不要** |
| レビュー | GitHub PRコメント | `PullRequests/*.md`でドキュメント |
| マージ | GitHubでマージボタン | ローカルでgit merge |

### 正しい作業手順

1. **必ず作業ブランチを作成してから作業を開始する**
2. **作業ブランチにコミットする**
3. **レビュードキュメントを作成する**（`PullRequests/issue-XX-review.md`）
4. **レビューを受けてmainにマージする**

---

## 基本フロー

**推奨**: 作業ブランチ（feature/*, fix/* 等）を作成してから実装を開始してください。

### 1. mainブランチの最新化

```bash
git checkout main
# プッシュは不要なので、ローカルのみ確認
```

### 2. ブランチの作成

```bash
git checkout -b feature/<task-name>
```

### 3. 開発・テスト

- コードの実装
- テストの作成・実行（`pytest tests/ -v`）
- リントチェック（`ruff check .`）

### 4. コミット

```bash
git add .
git commit -m "feat: 実装内容の説明"
# プッシュは不要
```

### 5. レビュードキュメント作成（開発完了の定義）

- `/pr-review <Issue番号>` スキルを実行
- または手動で `PullRequests/issue-XX-review.md` を作成

**⚠️ 重要**: レビュードキュメント作成までが「開発完了」です。

### 6. レビュー・修正

- レビュードキュメントの指摘に対応
- 指摘事項を修正

### 7. マージ

- レビュー承認後、main ブランチにマージ
- マージ後は作業ブランチを削除

---

## ブランチ命名規則

| プレフィックス | 用途 | 例 |
|--------------|------|-----|
| `feature/` | 新機能の追加 | `feature/add-new-component` |
| `fix/` | バグ修正 | `fix/timeout-handling-error` |
| `docs/` | ドキュメント変更 | `docs/update-readme` |
| `refactor/` | リファクタリング | `refactor/simplify-architecture` |
| `test/` | テスト関連 | `test/add-integration-tests` |

---

## 推奨されるプラクティス

- **mainブランチ保護**: mainブランチでの実装作業は避けてください
- **プッシュ不要**: このプロジェクトではGitHubへのプッシュは不要です。ローカルで完結します。

---

## コミットメッセージ規約（Conventional Commits）

```
<type>: <subject>

<body>

<footer>
```

### タイプ（type）

- `feat:` - 新機能の追加
- `fix:` - バグ修正
- `docs:` - ドキュメントのみの変更
- `refactor:` - リファクタリング（機能追加やバグ修正を含まない）
- `test:` - テストの追加・修正
- `chore:` - その他の変更（ビルドプロセス、ツール設定等）

### 例

```
feat: 新機能の追加

- 機能の説明
- テストを追加

Closes #3
```

## 開発前チェックリスト

**⚠️ 最重要**: 作業を開始する前に、必ず以下を確認してください。

- [ ] **現在のブランチが main ではないこと**
  ```bash
  # 確認コマンド
  git branch --show-current

  # main だった場合、即座に作業ブランチを作成してください
  git checkout -b feature/<task-name>
  ```
- [ ] **作業ブランチ（feature/*, fix/* 等）を作成している**
- [ ] テストがパスすることを確認（`pytest tests/ -v`）
- [ ] リントが通ることを確認（`ruff check .`）
- [ ] 実装内容の概要を理解している
- [ ] **プッシュは不要であることを理解している**

---

## 開発完了チェックリスト（レビュードキュメント作成時）

**開発完了の定義**: レビュードキュメントを作成してレビュー依頼を出した時点を「開発完了」とします。

- [ ] 新しい機能にはテストがある
- [ ] すべてのテストがパスする
- [ ] リントが通る
- [ ] コミットメッセージが規約に従っている
- [ ] レビュードキュメントを作成している（`/pr-review <Issue番号>`）
- [ ] ドキュメントの更新が必要な変更か確認
- [ ] 必要に応じてドキュメントを更新している
